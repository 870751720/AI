> 用户明确设定的行为边界与工作流程规则。每次会话必须加载并严格遵守。

## Git 工作流铁律

### 1. 提交与推送 —— 必须经用户明确授权

- **禁止**在未收到用户明确指令（如"提交"、"推送"）的情况下执行 `git commit` 或 `git push`
- 用户说"先提交"不等于授权提交，必须听到明确的"提交"/"push"字样

### 2. 提交后仓库必须干净

- 每次 `git commit` **前后**都必须检查 `git status`
- 确保提交后 working tree 无任何未跟踪文件、未提交修改或子模块残留变更
- 如有子模块变动，要么一并处理干净，要么在提交前还原，禁止留下脏状态

### 3. 规则落盘 —— 无需请示

- 用户明确的工作流规则、约束、偏好等，应立即写入文档落盘
- 写文件本身不需要用户授权，因为文件未进入 git 前用户可随时自行修改
- 只有 **commit / push** 动作需要等待明确指令
